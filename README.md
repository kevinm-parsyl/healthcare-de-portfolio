## Project 1 — Detail

### Clinical Context

This project models a diabetic patient population and calculates two CMS eCQMs:

**CMS122 — Diabetes: HbA1c Poor Control (>9%)**  
Measures the percentage of diabetic patients (ages 18–75) whose most recent HbA1c result during the measurement period exceeds 9.0%, or who had no HbA1c result. An inverse measure — lower rate indicates better diabetes management. National benchmark: 27.3% (CMS 2024 MIPS CQM data).

**CMS111 — Median ED Arrival to Departure Time for Admitted Patients**  
Measures median time in minutes between ED arrival and departure for patients subsequently admitted to the hospital. An efficiency measure — lower time indicates better throughput.

### Data

Synthetic FHIR R4 patient data generated by [Synthea](https://github.com/synthetichealth/synthea), a clinically realistic synthetic patient generator. Approximately 34,212 patients across Washington, Oregon, and California. No real PHI is involved.

Resource counts across the dataset:

| Resource Type | Count |
|---|---|
| Observation | 17,268,178 |
| Procedure | 5,405,264 |
| Encounter | 1,971,272 |

### Architecture

```
Synthea FHIR R4 bundles (JSON)
    ↓ Python ingestion pipeline
Snowflake internal stages (3 purpose-specific)
    ↓ COPY INTO
RAW schema (VARIANT columns, immutable)
    ↓ dbt staging models (views)
DEV_STAGING schema (flattened FHIR resources)
    ↓ dbt mart models (tables)
DEV_MART schema (dimensional model + CQM measures)
```

**Why VARIANT / raw-first:**  
FHIR bundles contain 20+ resource types in a single JSON file. Landing them as raw VARIANT preserves full fidelity, pushes all compute to Snowflake (disaggregated from storage), and enables reprocessing without re-ingestion. All decomposition via Snowflake's `LATERAL FLATTEN`.

**Why three stages:**  
Patient bundles, organization data, and practitioner data have different update frequencies, different owners, and different downstream consumers. Separate stages mirror real-world FHIR data contracts and map cleanly to FHIR R5 topic-based subscriptions (Project 2).

### Pipeline

`pipelines/fhir_ingestion.py` — Python ingestion pipeline connecting Snowflake stages to raw tables. Designed with each function mapping to a future Airflow task:

```python
get_stage_metadata()      → LIST stage, return file count and bytes
insert_manifest_record()  → Write PENDING record to LOAD_MANIFEST
copy_into_raw()           → Execute COPY INTO with METADATA$ columns
update_manifest_status()  → Update manifest to LOADED/PARTIAL/FAILED/SKIPPED
run_pipeline()            → Orchestrates all three stage/table pairs in order
```

Load ordering is enforced: organization → practitioner → patient. This respects referential integrity — encounter and condition resources reference organization and practitioner resources.

### dbt Models

**Staging layer (views):** One model per FHIR resource type. Flattens VARIANT bundles into typed relational columns using `LATERAL FLATTEN`. Aligned with QI-Core STU 6 profiles.

| Model | Source | Grain |
|---|---|---|
| stg_patient | RAW.BUNDLE | One row per patient |
| stg_encounter | RAW.BUNDLE | One row per encounter |
| stg_condition | RAW.BUNDLE | One row per condition |
| stg_observation | RAW.BUNDLE | One row per observation |
| stg_medication_request | RAW.BUNDLE | One row per prescription |

**Mart layer (tables):** Dimensional model with CQM measure models. Measure-specific clinical logic lives in CQM models only — fact tables contain general-purpose attributes.

| Model | Description |
|---|---|
| dim_patient | Patient dimension with VSAC-sourced diabetic cohort flag |
| fact_encounter | Encounter events with duration measures |
| fact_condition | Condition records with clinical status |
| fact_observation | Observation results including lab values |
| fact_medication_request | Prescription records |
| cms122_measure | CMS122 HbA1c poor control — one row per denominator patient |
| cms111_measure | CMS111 ED arrival to departure — one row per admitted ED encounter |

**Seeds:** Clinical terminology sourced from VSAC (Value Set Authority Center):

| Seed | Description | Rows |
|---|---|---|
| ref_value_sets | CMS eCQM value sets, May 2023 release (2024 performance year) | 105,272 |
| ref_code_system_map | FHIR URI to VSAC OID code system mapping | 6 |

Value set OIDs used:
- Diabetes: `2.16.840.1.113883.3.464.1003.103.12.1001` (370 ICD-10/SNOMED codes)
- HbA1c Laboratory Test: `2.16.840.1.113883.3.464.1003.198.12.1013` (5 LOINC codes)
- ED Evaluation and Management Visit: `2.16.840.1.113883.3.464.1003.101.12.1010` (6 CPT/SNOMED codes)

### Results

| Measure | Result | National Benchmark |
|---|---|---|
| CMS122 Poor Control Rate | 3.9% | 27.3% (CMS 2024) |
| CMS111 Median ED Time | 162 minutes | 60–300 minutes typical |

**Note on CMS122:** The 3.9% poor control rate is a known artifact of Synthea's synthetic data — HbA1c values are generated with an unrealistic distribution (mean ~5.2% vs expected ~7.5% for a diabetic population). Pipeline logic was independently validated as correct. See `notebooks/cms122_data_quality_analysis.ipynb` for full analysis including denominator validation, numerator pathway validation, and a simulation showing expected results (~28-32%) with a realistic HbA1c distribution. This finding demonstrates the analytical rigor to distinguish data quality issues from pipeline bugs.

### Tests

71 generic dbt tests (unique, not_null, accepted_values, relationships) across all staging and mart models. 1 singular test validating 1:1 relationship between VSAC value set names and OIDs.

```bash
dbt test          # run all tests
dbt build         # run models and tests together in DAG order
```

### Known Limitations & Production Considerations

**VSAC value sets:** Using CMS eCQM May 2023 release. Full production implementation would automate VSAC downloads on the annual release cycle and run `dbt seed` to update.

**is_superseded not implemented:** Raw tables include an `is_superseded` flag defaulting to FALSE. Supersession logic (marking prior bundles when a patient's updated bundle arrives) is deferred to Project 2's Airflow pipeline. All staging models include a TODO comment marking where `WHERE b.is_superseded = FALSE` should be added.

**Surrogate keys:** Source system UUIDs (Synthea-generated) are used as primary keys. In a multi-source production environment, surrogate keys generated via `dbt_utils.generate_surrogate_key()` would replace source identifiers to handle identity resolution across systems. See EMPI note below.

**EMPI:** Single data source eliminates the need for patient identity resolution. A production multi-source pipeline would require an Enterprise Master Patient Index (EMPI) to assign Global Patient Identifiers (GPIDs) before staging. The `identifiers_json` VARIANT column in `stg_patient` is preserved specifically to support this integration.

**CQL:** CQM logic is implemented in SQL rather than the CMS-standard Clinical Quality Language (CQL). For formal CMS reporting, measure logic should be validated against the official CQL definition using the Bonnie testing tool.

**Measurement period:** Hardcoded as 2024-01-01 to 2024-12-31. Project 3 will parameterize via dbt variables.

**QI-Core alignment:** Staging models are partially aligned with QI-Core STU 6 profiles. Known gaps: Encounter.diagnosis element extraction, no CQL implementation. Value sets sourced from the authoritative VSAC rather than hardcoded — the standard production pattern.

## Setup

**Prerequisites:** Python 3.11, Homebrew, Snowflake account (free trial sufficient)

```bash
# Clone and set up environment
git clone https://github.com/kevinm-parsyl/healthcare-de-portfolio
cd healthcare-de-portfolio
python3.11 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Configure Snowflake credentials
cp .env.example .env
# Edit .env with your Snowflake credentials

# Initialize dbt
cd healthcare_de_portfolio
dbt debug    # verify connection
dbt seed     # load VSAC value sets
dbt build    # run all models and tests
```

**Snowflake setup:** Run `snowflake/ddl/raw_tables.sql` and `snowflake/ddl/stages.sql` in your Snowflake worksheet before running the pipeline.

**Generating test data:** [Synthea](https://github.com/synthetichealth/synthea) required (Java 17+). See `scripts/dev_put_files.py` for uploading generated files to Snowflake stages.

## Healthcare Domain Notes

This portfolio is intentionally built on healthcare data to leverage domain expertise as a differentiator. The FHIR R4 standard, CMS eCQM framework, QI-Core profiles, and VSAC terminology are all production-relevant in healthcare data engineering roles. The limitations documented here (EMPI, CQL, VSAC automation) represent the delta between a portfolio demonstration and a production implementation — gaps that are well-understood and clearly articulated rather than unknown.
